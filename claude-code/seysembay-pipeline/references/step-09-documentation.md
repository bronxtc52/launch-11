# Шаг 9: CLAUDE.md + остальная документация

## Цель шага

Подготовить документацию **для самого Claude Code**, который будет писать код. Главный артефакт — `CLAUDE.md`, который Claude Code автоматически читает при каждом запуске.

## Главный принцип CLAUDE.md

**КОРОТКИЙ**. Максимум 60-100 строк. Это operational contract — правила игры, не полная документация.

Подробная документация (архитектура, ADR, спеки) — в отдельных файлах. CLAUDE.md ссылается на них.

См. также `claude-code-vibe-coding-guide.md` пользователя (раздел 3) — там расширенные правила и примеры. Не дублируй его, ссылайся.

## Workflow шага

### 1. Сборка CLAUDE.md

Структура файла:

```markdown
# <Название продукта>

## Что это
<2-3 предложения. Можно взять из Vision Story (шаг 2).>

## Stack
- Frontend: <технология, версия>
- Backend: <технология, версия>
- БД: <технология, версия>
- Хостинг: <Azure App Service / VM / ...>

## Команды
- Запуск dev: `npm run dev`
- Билд: `npm run build`
- Тесты: `npm run test`
- Линт: `npm run lint`
- Деплой: `<команда или ссылка на pipeline>`

## Структура репозитория
```
/src
  /pages       # Маршруты Next.js
  /components  # React-компоненты
  /lib         # Утилиты, клиенты API
  /db          # Миграции, схемы
/tests
/docs          # Документация (architecture, tech-spec)
/adr           # ADR
```

## Главные правила
- Всегда использовать TypeScript strict mode
- Всегда писать тесты для бизнес-логики (не для UI)
- Никогда не коммитить .env или секреты
- Все API-вызовы через клиент в /lib/api.ts (не fetch напрямую)
- Стилизация только через Tailwind classes (не CSS-modules)
- Использовать простейший подход — не переусложнять

## Не трогать
- /legacy/
- /generated/ (auto-generated)
- /vendor/

## Документация (когда нужно)
- Архитектура: /docs/architecture.md
- Tech spec: /docs/tech-spec.md
- Принятые решения: /adr/README.md
- FAQ: /docs/faq.md

## Контекст продукта (важно)
- Целевой пользователь: <одна фраза из блока B>
- Northern Star: <одна фраза из шага 1>
- Что НЕ делаем: <3-5 пунктов из блока D>
```

**Важные сокращения от 100 строк:**

- Не вставляй полные сценарии — они в tech-spec
- Не вставляй полные правила code style — добавь линтер с конфигом
- Не вставляй имена контактов команды — в README

### 3. Финализация документации

Файлы для будущего репозитория:

| Файл в репо | Источник |
|-------------|----------|
| `CLAUDE.md` | `09-CLAUDE.md` (из этого шага) |
| `README.md` | `09-README-for-repo.md` |
| `docs/architecture.md` | `08-architecture.md` |
| `docs/tech-spec.md` | `08-tech-spec.md` |
| `docs/vision.md` | `06-vision-contract-final.md` |

| `adr/` | вся папка как есть |

**Важно:** в нашей рабочей папке мы держим всё под префиксами `01-`, `02-`, ..., `09-` — это **наша история**, чтобы видеть прогресс. При **переезде в репозиторий** префиксы убираются, файлы переименовываются.

Скилл должен **отдельным разделом** в `_pipeline-status.md` показать пользователю карту переименования файлов:

```
СТРУКТУРА В РАБОЧЕЙ ПАПКЕ → СТРУКТУРА В РЕПОЗИТОРИИ

01-northern-star.md         → docs/vision/northern-star.md (опционально)
02-vision-story.md          → docs/vision/story.md (опционально)
03-vision-contract-draft.md → (не переезжает, остаётся как история)
04-research-brief.md        → (не переезжает)
05-research-results.md      → (не переезжает)
06-vision-contract-final.md → docs/vision.md
06-faq.md                   → docs/faq.md
07-story-sync-check.md      → (не переезжает)
08-architecture.md          → docs/architecture.md
08-tech-spec.md             → docs/tech-spec.md
09-CLAUDE.md                → CLAUDE.md (корень)
09-README-for-repo.md       → README.md (корень)

adr/                        → adr/
```

### 4. Memory-слой для Claude Code (опционально, для продвинутых)

Если пользователь работает в Claude Code и хочет следовать паттерну из `claude-code-vibe-coding-guide.md` (раздел 3.5), можно разделить:

```
.claude/rules/
├── memory-profile.md      # Факты о проекте
├── memory-preferences.md  # Как любим
├── memory-decisions.md    # ADR ссылки
└── memory-sessions.md     # История работы
```

Это **необязательно**. Для большинства проектов хватит одного CLAUDE.md.

## Шаблон артефакта

См. также `templates/09-CLAUDE.md` — пример CLAUDE.md, готового к копированию.

## Антипаттерны

- **CLAUDE.md на 500 строк.** Не помещается в контекст эффективно. Делай 100 строк, остальное — в `/docs`.
- **Inline всей документации.** Если в CLAUDE.md процитировал tech-spec — ты дублируешь информацию. Ссылайся: "См. /docs/tech-spec.md".
- **Забыть про .gitignore и .env.example.** Это базовая гигиена, но часто забывается на этапе документации.
- **README без Quick Start.** Если разработчик не может за 30 секунд понять как запустить — README плохой.

## Чек-лист содержания CLAUDE.md

- [ ] Что это (1-2 предложения)
- [ ] Stack (4-6 строк)
- [ ] Команды (4-6 команд)
- [ ] Структура репо (5-15 строк)
- [ ] Главные правила (3-7 правил)
- [ ] Что не трогать (2-5 директорий)
- [ ] Ссылки на документацию (3-5 ссылок)
- [ ] Контекст продукта (Northern Star + Целевой пользователь + Не делаем)

Если все галочки и не больше 100 строк — отлично.
