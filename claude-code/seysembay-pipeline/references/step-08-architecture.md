# Шаг 8: Architecture + Tech-spec + ADR (технические)

## Цель шага

Перевести финальный Vision Contract в **техническую спецификацию**: стек, структура, БД, хостинг, интеграции. Все значимые технические решения → ADR.

## Главный принцип

Не выбирай за пользователя. Предлагай **2-3 варианта на каждое значимое решение**, фиксируй выбор как ADR. То же правило, что в шаге 5.

## Особый момент — Azure-карта

⚠️ **Если пользователь разворачивает в Azure** — обязательно проверь `azure-resources-mapping.md` (если файл в контексте проекта).

У пользователя **297 ресурсов в 88 resource groups**. Это значит:

1. **Перед предложением "создать новую БД"** — посмотри, нет ли подходящего MySQL Flexible / SQL Server, на котором можно создать ещё одну БД. Особо: сервер `marinehealth` (MySQL Flexible, Germany West Central) или `24exchangekz` (SQL Server, North Europe).

2. **Перед предложением "создать App Service"** — проверь существующие App Service Plans (их 8). Можно ли добавить ещё один сайт на существующий план?

3. **Перед предложением "создать VM"** — есть ли подходящая существующая? Можно ли модулем добавить на текущую инфраструктуру?

4. **Регион по умолчанию — North Europe** (Dublin). 42% ресурсов уже там. Если нет особой причины — не разъезжаемся.

5. **Не предлагай создавать новый Bastion** — у пользователя уже 3, каждый ~$140/мес. Используй существующий через peering или JIT.

**Если файла azure-resources-mapping.md нет в контексте** — спроси у пользователя: "Будем разворачивать в Azure? Если да — у тебя есть существующая инфраструктура?"

### 3. По каждому крупному решению — варианты

**Список крупных технических решений (обычно):**

| Решение | Когда обсуждать |
|---------|-----------------|
| Frontend стек (React/Vue/Next.js/Astro/...) | Если есть UI |
| Backend стек (Node/Python/Go/.NET/...) | Если есть API |
| База данных (PostgreSQL/MySQL/MongoDB/...) | Всегда |
| Авторизация (свой/Auth0/Azure AD/Firebase/...) | Всегда |
| Хостинг (Azure App Service/VM/контейнеры/serverless) | Всегда |
| Storage (Azure Blob/S3/Files) | Если есть медиа/файлы |
| Платежи (Stripe/local провайдер/...) | Если есть монетизация |

### 5. Технические ADR

- ...

**Особо важные ADR для Azure-проектов:**
- ADR про выбор региона (если не North Europe — нужно обоснование)
- ADR про переиспользование существующих ресурсов vs создание новых
- ADR про SKU/tier (Basic/Standard/Premium)

### 6. Если стек существующий

Если пользователь делает модуль/расширение существующего проекта (например, новый сервис для Marine Health):
- Не предлагай 3 разных стека если у проекта уже есть стек
- Унаследуй стек из существующего проекта, зафиксируй в ADR
- Обсуди только новые решения (новая БД? новые интеграции?)

## Антипаттерны

- **Выбор технологии "потому что модно"** без альтернатив. Зафиксируй альтернативы.
- **Игнорирование существующей инфраструктуры.** Создал новый VM, когда было свободное место на старом.
- **Региональное разнесение без причины.** Если основная инфраструктура в North Europe — клади туда же, не в West Europe.
- **Технические решения без ADR.** Через 6 месяцев забудешь почему MySQL, а не PostgreSQL.
- **Слишком сложная архитектура на старте.** v1 — это монолит. Микросервисы можно в v3, когда будет понимание реальных нагрузок.
