# Шаг 10: Task decomposition (BACKLOG)

## Цель шага

Разбить весь Tech Spec на **конкретные задачи по 1-3 часа** работы Claude Code. На выходе — `BACKLOG.md` с задачами, готовыми к запуску.

## Главные принципы

### 1. Размер задачи — 1-3 часа

Если задача больше — это **не задача, а тема**. Разбивай.

**Хорошие задачи:**
- "T-007: Создать API endpoint GET /api/partners/:id/dashboard"
- "T-012: Добавить миграцию для таблицы orders"
- "T-018: Реализовать компонент <PartnerLevelCard/>"

**Плохие задачи (это темы, не задачи):**
- "Сделать аутентификацию" — это 5-10 задач
- "Запилить дашборд" — это 8-15 задач
- "Интегрировать с 1С" — это отдельный эпик

### 2. Задачи должны быть независимы

Где возможно — задачи можно запускать **параллельно** (через git worktrees, см. `claude-code-vibe-coding-guide.md` раздел 7).

Зависимости — это нормально, но укажи их явно: "Зависит от T-003".

### 3. У каждой задачи — Definition of Done

Без чёткого DoD задача "никогда не выполнена". Минимум:
- Тесты проходят
- Линтер чистый
- Сделано конкретно что (1-2 предложения)

## Workflow шага

### 1. Прочитай предыдущие шаги

- `06-vision-contract-final.md` — что строим
- `08-architecture.md` — на каком стеке
- `08-tech-spec.md` — какие модули, эндпоинты, схема БД
- Все ADR

### 2. Группируй задачи по эпикам

Сначала верхний уровень — эпики (большие куски). Потом каждый эпик → задачи.

Типичные эпики для web-проекта:

| Эпик | Примерное количество задач |
|------|----------------------------|
| Setup (репо, CI/CD, базовая структура) | 3-5 |
| Infrastructure (БД, миграции, deploy) | 5-10 |
| Authentication (регистрация, логин, сессии) | 5-10 |
| Module: <Главная фича> | 10-20 |
| Module: <Вторая фича> | 10-20 |
| Testing & QA | 5-10 |
| Polish (UX, тексты, мелочи) | 5-15 |

### 3. По каждому эпику — задачи

Шаблон задачи:

```markdown
### T-NNN: <название>

**Эпик:** <название эпика>
**Размер:** S (≤1 час) / M (1-2 часа) / L (2-3 часа)
**Зависимости:** T-XXX, T-YYY (если есть)
**Можно параллельно с:** T-ZZZ (опционально)

**Что сделать:**
- Шаг 1
- Шаг 2
- Шаг 3
(3-7 пунктов, конкретно)

**Definition of Done:**
- [ ] <критерий 1>
- [ ] <критерий 2>
- [ ] Тесты проходят (минимум N кейсов)
- [ ] `npm run lint` без ошибок
- [ ] (Опционально) Документация обновлена

**Связано с ADR:** ADR-NNN (если применимо)
**Файлы, которые затронет:**
- src/...
- tests/...

**Подсказки для Claude Code:**
(опционально — если есть нюансы, на которые надо обратить внимание)
```

### Пример задачи

```markdown
### T-007: API endpoint GET /api/partners/:id/dashboard

**Эпик:** Module: Dashboard
**Размер:** M (1-2 часа)
**Зависимости:** T-003 (миграция БД), T-005 (auth middleware)
**Можно параллельно с:** T-008 (frontend компонент)

**Что сделать:**
- Создать роут в `src/routes/partners.ts`
- Реализовать запрос в БД, который возвращает:
  - Текущий уровень партнёра
  - Объём за месяц (в денежной форме)
  - Прогноз до следующего уровня
- Добавить валидацию: только сам партнёр или его спонсор может видеть данные
- Покрыть юнит-тестами: минимум 3 кейса (успешный запрос, попытка чужих данных, БД-ошибка)
- Документировать в OpenAPI (`docs/openapi.yaml`)

**Definition of Done:**
- [ ] Эндпоинт отвечает за <200мс на dev-данных
- [ ] Тесты проходят: 3+ кейсов
- [ ] OpenAPI обновлён
- [ ] Авторизация работает корректно
- [ ] `npm run lint` без ошибок

**Связано с ADR:** ADR-009 (auth), ADR-011 (API структура)

**Файлы, которые затронет:**
- src/routes/partners.ts
- src/services/dashboard.ts
- tests/api/partners.test.ts
- docs/openapi.yaml

**Подсказки для Claude Code:**
- Используй существующий `getPartnerById()` в src/services/partners.ts
- Прогноз считается через `calculateNextLevelForecast()` — функция уже есть
```

## Антипаттерны

- **Задача "Реализовать модуль X".** Это не задача, это эпик. Разбей.
- **Задачи без DoD.** Без критерия "выполнено" задача будет тлеть бесконечно.
- **Игнорирование зависимостей.** Если T-005 зависит от T-003, а ты не указал — Claude Code начнёт T-005 первым и упадёт.
- **Слишком детальные задачи.** Не нужно расписывать каждую строчку кода. Claude Code сам разберётся в деталях. Задача — "что", а не "как".
- **Слишком крупные оценки.** Если задача оценена в "1 день" — это значит ты не разбил. 1 день = 3-5 задач.
