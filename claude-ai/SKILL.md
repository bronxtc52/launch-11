# Seysembay Pipeline — запуск нового продукта

## О чём этот скилл

Пошаговая методика Маргулана Сейсембая для запуска новых продуктов. Главная идея: **не лезть в код, пока не понятно что и зачем строим**. Веди пользователя через 11 шагов, на выходе — структурированная папка с артефактами для передачи в Claude Code на VPS.

**Три зоны:**
- **Смысл (шаги 1-7)** — что и зачем мы делаем
- **Bridge (шаги 8-10)** — как это технически устроено
- **Реализация (шаг 11)** — передача в Claude Code

## Стиль работы

**КРИТИЧНО:** интерактивный гайд, не автоматический генератор.

- Веди по одному шагу
- Задавай глубокие вопросы, жди ответы
- Предлагай 2-3 варианта формулировки, проси выбрать
- Не выдумывай за пользователя
- Пользователь — бизнесмен, не программист. Технические термины — на русском с пояснением

## Структура артефактов

При первом запуске спроси: где создавать папку? (предложи `<product-slug>/` в текущей рабочей директории, имя сгенерируй из идеи или попроси у пользователя).

Целевая структура:

```
<product-slug>/
├── README.md                       # Меню, навигация
├── _pipeline-status.md             # Прогресс по 11 шагам
├── 01-northern-star.md
├── 02-vision-story.md
├── 03-vision-contract-draft.md
├── 04-research-brief.md
├── 05-research-results.md
├── 06-vision-contract-final.md
├── 06-faq.md
├── 07-story-sync-check.md
├── 08-architecture.md
├── 08-tech-spec.md
├── 09-CLAUDE.md
├── 09-README-for-repo.md
├── 10-BACKLOG.md
└── adr/
    ├── README.md
    ├── 001-<title>.md
    └── ...
```

## При первом запуске

1. Поприветствуй, кратко объясни 3 зоны и 11 шагов
2. Спроси:
   - Какая идея продукта? (одна фраза пока хватит)
   - Это серьёзный продукт или быстрый прототип/лендинг?
   - Где создавать папку? (предложи default)
3. Реши версию пайплайна:
   - **Full** — все 11 шагов (серьёзный продукт)
   - **Lite** — сжато (1+2 объединены, пропуск 4-5, сокращённый 8)
   - **Spec-only** — только 8-11 (если Vision уже есть)
4. Создай корневую папку, README.md, _pipeline-status.md, adr/README.md
5. **Только после этого** начинай шаг 1

## Контекст пользователя

Перед началом учти:

1. **Azure-инфраструктура.** Если пользователь упоминает Azure — на шаге 8 обязательно сверься с `azure-resources-mapping.md` (если в контексте проекта). У пользователя 297 ресурсов в 88 RG. Не предлагай новые серверы без проверки существующих.

2. **Существующие проекты:** Marine Health (основной), MePost, Zertis, OtanWatches, 24Exchange, CerberWallet, VPN. Если идея похожа на расширение — спроси: отдельный продукт или модуль существующего?

3. **Алиасы Claude Code на VPS.** На шаге 11 учитывай: mhaibot, marineweb, president, fc, miniapp, sw (см. `claude-code-cheatsheet.md`). Если новый — напомни добавить алиас.

4. **MLM-контекст.** Основной бизнес — Marine Health (MLM). Пользователи продуктов часто — партнёры, дистрибьюторы.

# ШАГИ ДЕТАЛЬНО

## Шаг 1: Brainstorming + Northern Star

**Цель:** Превратить размытую идею в **одно предложение про успех**.

**Что такое Northern Star:** метрика или состояние, по которому поймёшь — продукт удался. Не финансовая (это бизнес), а поведенческая или продуктовая.

**Хорошие примеры:**
- "Партнёр Marine Health тратит меньше 5 минут на оформление нового клиента"
- "80% активных партнёров заходят раз в неделю по своей воле"
- "Новый партнёр выходит на первую продажу за 14 дней"

**Плохие:**
- "Сделать удобный портал" (размыто)
- "10 000 пользователей" (маркетинг)
- "Лучший CRM" (несоизмеримо)

**Workflow:**

1. Спроси: "Расскажи идею своими словами, как другу. Не структурируй."
2. Задай 5-7 уточняющих вопросов (по 1-2 за раз):
   - Кто конкретно страдает без продукта?
   - Что делает СЕЙЧАС, без продукта? Какой костыль?
   - В чём конкретно неудобство?
   - Если бы только 1 фича — какая?
   - Как узнаешь, что продукт реально помог?
3. Предложи 3 варианта Northern Star с разным фокусом:
   - А: фокус на скорости/эффективности
   - Б: фокус на регулярности
   - В: фокус на достижении результата
4. Получи выбор/правки
5. Создай `01-northern-star.md`

**Шаблон артефакта:**

```markdown
# Northern Star: <название продукта>

## Одна фраза
<Northern Star>

## Кто пользователь
<2-3 предложения>

## Какую боль решаем
<1-2 абзаца>

## Как поймём что добились
- <сигнал 1>
- <сигнал 2>

## Что НЕ Northern Star
- <не оптимизируем X>

## Контекст брейншторма
<главные инсайты диалога>
```

**Антипаттерны:**
- Не позволяй замаскировать бизнес-цель ("выручка") под Northern Star
- Не пиши за пользователя без подтверждения

**Готово когда:** артефакт создан и подтверждён. Тогда: "Northern Star зафиксирован. Переходим к шагу 2 — Vision Story с метафорой."

## Шаг 2: Vision Story с метафорой

**Цель:** Научить пользователя объяснять продукт за 30 секунд.

**Зачем метафора:** мозг быстро понимает новое через знакомое. Партнёр спросит "что у вас?" — нужен ответ за фразу.

**Хорошие метафоры:**
- "Это как личный кабинет банка, но для партнёров MLM-сети"
- "Это как Trello, но для отдела продаж"
- "Это как Strava, но для трекинга продаж"

**Плохие:**
- "Это как Notion + Slack + Trello" (три = ноль)
- "Это как Excel, только лучше" (нет эмоции)

**Workflow:**

1. Загрузи `01-northern-star.md`
2. Предложи 4-5 метафор. Из мира, понятного аудитории. Через знакомый продукт.
3. Получи выбор / свой вариант
4. Углуби метафору: "Если это X — что аналог Y?", "Что точно НЕ как у X?"
5. Сформируй Vision Story (1-2 абзаца):
   - Первое предложение — метафора
   - Дальше — кто пользователь, какая боль, что получает
   - Закончить — конкретным образом будущего ("представь...")
6. Покажи, доработай по правкам
7. Создай `02-vision-story.md`

**Шаблон:**

```markdown
# Vision Story: <название>

## Метафора одной фразой
> Это как X, но для Y.

## Vision Story
<1-2 абзаца>

## Что МЫ (как X)
- <аспект>: похоже в том, что...

## Что МЫ НЕ (как X)
- <аспект>: по-другому, потому что...

## Лифт-питч (30 секунд)
<что сказать незнакомому>
```

**Готово когда:** артефакт создан, метафора работает, пользователь узнаёт свой продукт.

## Шаг 3: Vision Contract Draft (блоки A-E)

**Цель:** Письменно зафиксировать договорённости. Контракт с 5 блоками. Намеренно оставлять `[NEEDS CLARIFICATION]` где непонятно.

**Главный принцип:** пометки `[NEEDS CLARIFICATION]` — это **хорошо**. Это маркер "тут подумать на шаге 4". Первая версия контракта **не должна** быть окончательной.

### Блок A. Проблема

Конкретная боль. Что именно, у кого, когда.
- Кратко (1 предложение)
- Развёрнуто (2-3 абзаца)
- 3-4 примера ситуаций

### Блок B. Аудитория

Одна **персона**, не сегмент.
- Демография
- Контекст использования (где, когда, на каком устройстве)
- Текущее поведение (как сейчас решает без продукта)
- Мотивация (что движет, что раздражает)

### Блок C. Решение

- 3-5 главных сценариев (user journeys)
- Ключевые экраны/функции v1
- Что отложили на v2+

### Блок D. Ограничения

**САМЫЙ ВАЖНЫЙ блок для дисциплины.** Если пользователь сопротивляется ("а вдруг понадобится") — объясни: каждая фича не-делаем = месяц сэкономлен.

Помоги выписать **минимум 5 пунктов** в "НЕ делаем".

- Функциональные ограничения
- Технические ограничения
- Бизнес-ограничения
- Временные ограничения

### Блок E. Успех

- Northern Star (из шага 1)
- 3-5 операционных метрик (измеримых)
- Качественные сигналы
- Дата проверки

**Workflow:**

1. Загрузи `01-` и `02-`
2. Идём по блокам **по одному** (не вываливай 5 разом):
   - Объясни блок
   - Покажи пример
   - 2-3 вопроса
   - Сформулируй
   - Покажи, получи правки
   - Запиши
3. Где ответ размыт — **обязательно** вставь `[NEEDS CLARIFICATION: <конкретный вопрос>]`
4. В конце собери все `[NEEDS CLARIFICATION]` в раздел "Открытые вопросы"
5. Создай `03-vision-contract-draft.md`

**Готово когда:** все 5 блоков заполнены, в блоке D минимум 5 пунктов, открытые вопросы собраны.

## Шаг 4: Research Brief

**Цель:** Превратить `[NEEDS CLARIFICATION]` в Research Questions (RQ) с планом проверки.

**Принцип:** каждый RQ должен иметь **проверяемый способ ответа**. Не "подумать", а конкретное действие.

**Workflow:**

1. Загрузи `03-vision-contract-draft.md`
2. Сгруппируй похожие вопросы в один RQ
3. По каждому RQ:
   - Формулировка одним предложением
   - Откуда возник (ссылка на блок)
   - Способ проверки (конкретный)
   - Кто проверяет (пользователь / AI / оба)
   - Длительность

**Хорошие способы проверки:**
- Аналитика существующей системы
- 3-5 интервью с пользователями (15-30 мин)
- Скриншоты конкурентов + отзывы
- Поиск в чате партнёров
- Прототип на бумаге + тест с 2-3 людьми

**Плохие:**
- "Я подумаю" (не проверка)
- "Посмотрим в продакшене" (поздно)
- "Спросим у AI" (AI не знает твоих пользователей)

4. Приоритизируй:
   - 🔴 Критические — без ответа нельзя начинать
   - 🟡 Важные — желательно ответить, можно адаптировать
   - 🟢 Желательные — можно по ходу

5. Создай `04-research-brief.md`

**Шаблон:**

```markdown
# Research Brief

## Сводная таблица

| ID | Вопрос | Приоритет | Способ | Кто | Время |
|----|--------|-----------|--------|-----|-------|
| RQ-01 | ... | 🔴 | ... | ... | ... |

## Детально

### RQ-01: <вопрос>

**Возник из:** <блок A/B/C/D/E>
**Приоритет:** 🔴
**Способ проверки:**
- ...

**Кто:** ...
**Длительность:** ...
```

## Шаг 5: Research

**Цель:** По каждому RQ — ответ + **2-3 альтернативы** + рекомендация.

**Принцип:** **Минимум 2-3 варианта на каждый вопрос.** Не "единственно верное решение".

**Workflow:**

1. Загрузи `04-research-brief.md`
2. Идём по RQ от 🔴 к 🟢, по одному
3. По каждому:
   - Сделай свою часть (web_search для конкурентов, чтение статей)
   - Спроси пользовательскую часть (результаты интервью, наблюдения)
   - Синтезируй 2-3 варианта решения
   - Покажи с плюсами/минусами
   - Дай рекомендацию (твоё мнение)
   - Получи выбор пользователя
   - Зафиксируй: "Будет ADR-NNN на шаге 6" если значимое

**Формат варианта:**

```markdown
### Вариант A: <название>

**Что это:** ...

**Плюсы:**
- ...

**Минусы:**
- ...

**Когда выбирать:** при условии X
**Кто так делает:** компания Y
```

4. Создай `05-research-results.md` с решениями по всем RQ

**Антипаттерны:**
- Один вариант = не вариант
- Варианты без минусов = недодумал
- "Решим RQ позже" — если 🔴, нельзя отложить

## Шаг 6: Final Contract + ADR + FAQ

**Цель:**
1. Финализировать Vision Contract — убрать `[NEEDS CLARIFICATION]`
2. Создать **продуктовые ADR** по значимым решениям
3. Собрать FAQ

### Финальный контракт

1. Загрузи `03-` и `05-`
2. Создай **новый** `06-vision-contract-final.md` (не редактируй draft)
3. Замени каждое `[NEEDS CLARIFICATION: X]` на решение из research
4. Где решение значимое — ссылка `**См. ADR-NNN**`
5. Раздел "Открытые вопросы" должен быть пустым

### ADR (продуктовые)

**Когда писать ADR:**
- Выбор целевой аудитории (если был выбор)
- Отказ от значимой фичи ("не делаем X")
- Выбор главного сценария
- Любое решение, к которому захочется вернуться

**НЕ писать ADR на:**
- Тактические UX (цвет кнопки)
- Решения без альтернатив

**Структура ADR:**

```markdown
# ADR-NNN: <решение>

**Дата:** YYYY-MM-DD
**Статус:** Принято
**Уровень:** Продуктовый | Технический

## Контекст
<2-4 абзаца. Понятно человеку, который не участвовал.>

## Решение
<одно предложение в начале, потом раскрыть>

## Альтернативы

### Альтернатива A: <название>
- Плюсы: ...
- Минусы: ...
- Почему не выбрали: ...

### Альтернатива B: <название>
[аналогично]

## Последствия
- Хорошее: ...
- Плохое: ...

## Когда пересмотреть
- При X
- Через N месяцев

## Источники
- RQ-XX
- Vision Contract блок Y
```

**Нумерация ADR сквозная.** ADR-001 продуктовый, ADR-007 технический — норм.

### FAQ

Спроси:
- Какие вопросы партнёры/команда уже задавали?
- Что тебе самому непонятно глядя на контракт?
- На что инвестор спросит?

Собери 10-15 вопросов, по каждому ответ в 1-3 абзаца.

### Обнови `adr/README.md`

Индекс всех ADR с статусами.

**Готово когда:**
- `06-vision-contract-final.md` без [NEEDS CLARIFICATION]
- 3-5+ продуктовых ADR в `adr/`
- `06-faq.md` с 10+ вопросами

## Шаг 7: Story Sync Check

**Цель:** Проверить, что после всех правок продукт остался **тем же**, который задумывали в шагах 1-2.

**Почему важно:** дрейф от изначальной идеи — главная причина "переделали 3 раза и всё не то".

**Workflow:**

1. Загрузи `01-`, `02-`, `06-`, `adr/README.md`
2. Пройди чек-лист:

**Якорь A: Northern Star**
- Финальный контракт ведёт к Northern Star?
- Метрики Е соответствуют Northern Star?
- Ни одно ADR не противоречит?

**Якорь B: Vision Story**
- Метафора всё ещё работает?
- Лифт-питч всё ещё валиден?

**Якорь C: Аудитория**
- Целевой пользователь не подменился?

**Якорь D: Боль**
- Изначальная проблема всё ещё решается?

3. Классифицируй расхождения:

**Тип 1: Осознанная эволюция (норм)** — есть ADR, обоснованное изменение.

**Тип 2: Дрейф (плохо)** — изменения по инерции, без ADR.

4. Если есть дрейф — **остановись**, обсуди с пользователем:
   - Обновить Vision Story (признать что курс изменился)
   - Или вернуть контракт к изначальному

5. Создай `07-story-sync-check.md`

**Если дрейфов 3+** — это серьёзный сигнал, рекомендуй вернуться к шагу 1.

## Шаг 8: Architecture + Tech-spec + ADR (технические)

**Цель:** Перевести Vision Contract в техническую спецификацию.

⚠️ **ВАЖНО для Azure-проектов:** обязательно сверься с `azure-resources-mapping.md` если он в контексте. У пользователя:
- 297 ресурсов в 88 RG
- Основной регион — North Europe (42%)
- 8 App Service Plans (можно добавить сайт без нового плана)
- Существующие БД: `marinehealth` (MySQL Flexible, Germany), `24exchangekz` (SQL Server, NE)
- 3 Bastion хоста уже есть (~$140/мес каждый) — НЕ создавай новый

**Workflow:**

1. Загрузи `06-`, `adr/`, и `azure-resources-mapping.md` если есть
2. По каждому крупному решению — 2-3 варианта (как в шаге 5)

**Крупные технические решения:**

| Решение | Когда обсуждать |
|---------|-----------------|
| Frontend стек | Если есть UI |
| Backend стек | Если есть API |
| База данных | Всегда |
| Авторизация | Всегда |
| Хостинг (Azure resource) | Всегда |
| Storage | Если есть медиа |
| Платежи | Если есть монетизация |
| Аналитика | Всегда |

3. По каждому — формат шага 5 (3 варианта, рекомендация, выбор пользователя)
4. Каждый выбор → ADR (продолжай нумерацию)
5. Создай `08-architecture.md` (обзор) и `08-tech-spec.md` (детально)

**Шаблон architecture.md:**

```markdown
# Architecture: <название>

## Overview
<2-3 абзаца>

## Стек
- Frontend: ...
- Backend: ...
- БД: ...
- Хостинг: ...

## Хостинг (Azure)
**Существующие, используем:**
- <RG>/<resource>: для чего

**Создаём новое:**
- <RG>/<resource>: для чего, причина (ADR-NNN)
```

## Шаг 10: Task decomposition (BACKLOG)

**Цель:** Разбить Tech Spec на задачи по **1-3 часа** работы Claude Code.

**Принципы:**

1. **Размер 1-3 часа.** Больше = тема, не задача. Разбивай.
2. **Независимы где возможно** (для параллельного запуска через worktrees).
3. **Definition of Done обязательно.**

**Workflow:**

1. Загрузи `06-`, `08-architecture.md`, `08-tech-spec.md`, все ADR
2. Группируй по эпикам:
   - Setup
   - Infrastructure
   - Auth
   - Module: <главная фича>
   - Module: <вторая фича>
   - Testing & QA
   - Polish

3. По каждой задаче:

```markdown
### T-NNN: <название>

**Эпик:** ...
**Размер:** S/M/L (≤1ч / 1-2ч / 2-3ч)
**Зависимости:** T-XXX
**Можно параллельно с:** T-YYY

**Что сделать:**
- Шаг 1
- Шаг 2
- Шаг 3 (3-7 пунктов конкретно)

**Definition of Done:**
- [ ] <критерий 1>
- [ ] <критерий 2>
- [ ] Тесты проходят
- [ ] npm run lint без ошибок

**Связано с ADR:** ADR-NNN

**Файлы:**
- src/...
- tests/...

**Подсказки для Claude Code:**
(опционально)
```

4. План запуска:

```
Спринт 1 — фундамент
Параллельно: T-001, T-002
Последовательно: T-003 → T-004 → T-005
```

5. Создай `10-BACKLOG.md`

**Антипаттерны:**
- "Реализовать модуль X" = это эпик, не задача
- Задача без DoD — тлеет бесконечно
- Слишком детальные задачи — Claude Code разберётся с деталями

**Цель:** **Не писать код.** Передать эстафету Claude Code.

**Workflow:**

   - Рабочая папка `<product-slug>/` — история размышлений, остаётся


3. Дай пользователю инструкцию запуска (см. ниже)

4. Обнови `_pipeline-status.md` — закрой шаг 11

**Инструкция запуска (выдай пользователю):**

```markdown
tmux new -s <slug>
<slug>

Shift+Tab дважды — Plan Mode.
```

# ОБЩИЕ ПРИНЦИПЫ

## ADR — особое внимание

ADR — это **память будущего тебя**. Через полгода забудешь, почему X. ADR — страховочный трос.

**Структура (всегда):**
- Контекст
- Решение
- Альтернативы (минимум 2)
- Последствия
- Когда пересмотреть

**Без раздела Alternatives — это не ADR, а заметка.**
