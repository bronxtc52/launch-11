# План v2: возврат права на прогон + переживание перегрузки Claude

ТЗ: `entitlement-refund-and-overload-retry.md`. Класс **M**, домены: деньги, прод-секреты.

> **v1 отправлен в мусор ревьюером.** Три несущих утверждения не пережили контакта с кодом:
> порядок блокировок был обратен собственной прозе; артефакт-гард не ловил гонку, ради которой
> вводился, и отказывал в возврате на сценарии самого инцидента; бюджет ретраев множился на
> tool-loop в ~48 минут. Ниже — переделка, а не косметика. Находки помечены `[C1]`, `[M2]` и т.д.

## 0. Новый инвариант, который надо назвать вслух

**Строку сессии нельзя удалять — она реестр списания.** На этом держится вся идемпотентность
возврата. Не назовём — следующий человек вернёт `DELETE` и молча откроет дыру обратно.
После реализации — строкой в `CLAUDE.md`.

## 1. Миграция `006_entitlement_ledger.sql`

  ADD COLUMN IF NOT EXISTS consumed TEXT NOT NULL DEFAULT 'none',
  ADD COLUMN IF NOT EXISTS refunded BOOLEAN NOT NULL DEFAULT false;

ALTER TABLE sessions
  ADD CONSTRAINT sessions_consumed_chk CHECK (consumed IN ('free','paid','none'));  -- [m3]

  Решение: берём `abandoned`, `aborted` вычищаем из докстринга, чтобы словарь не рос сам собой `[m4]`.
- **Бэкфилл `[m3]`:** перед накатом ещё раз проверить `SELECT count(*) FROM sessions WHERE

  Пусто → бэкфилл не нужен; не пусто → одноразовый `UPDATE ... SET consumed='free'` для них.

## 3-bis. Доставка спеки идемпотентна `[C-1]` — иначе чиним инцидент, сдвинув его вправо

**Самая опасная находка v2.** `finish` ставит `finished` и коммитит, а спека уходит в
`on_document` (`turn.py:222`) **после**. Telegram чихнул (flood-control, блип, спека до 200 КБ) →
спеки нет; `reset` не вернёт (`WHERE status='active'`); `resume` не найдёт (фильтр по `active`) →
`/start` создаёт новую сессию → `free_used=1` → **пейволл**. А `finish` навсегда отвечает
`StepError("сессия уже завершена")` — и условный UPDATE из §3 делает этот гард **персистентным**:
спека недостижима вообще никогда.

Тот же инцидент, сдвинутый на 30 секунд вправо: человек заплатил за наш сбой.

**Принцип: статус гейтит ДЕНЬГИ, а не выдачу.**

- Порядок внутри `finish` фиксируем жёстко: `can_finish` → `assemble` → **условный UPDATE** →
  `return spec`. Обратный порядок (UPDATE до `assemble`) даёт `finished` без спеки и без возврата.
- `finish` на уже `finished` сессии — **пересобирает и отдаёт спеку снова**, а не бросает
  `StepError`. Артефакты живы, `assemble` детерминирован, деньги уже списаны — отдавать безопасно.
  Гард `orchestrator.py:191` снять.
- **Новая команда `/spec`** — re-deliver из артефактов: единственный путь для того, у кого
  доставка упала.
- `on_document` обернуть: провал доставки → notice «спека готова, пришли /spec», не generic «Упс».

## 3. `finish` обязан стать условным `[C1]` — это дыра в деньгах

Сейчас `set_status` (`pg_repo.py:113`) шлёт `WHERE id=$1` без проверки статуса, а защита
`if session.status == "finished"` (`orchestrator.py:191`) живёт **на объекте в памяти** — между
двумя параллельными хендлерами её нет.

Сценарий утечки: идёт ход → человек жмёт «начать заново» → возврат прошёл (`free_used=0`) →
ход доходит до `finish` → перезаписывает `abandoned` обратно в `finished` и **отдаёт spec.md**.
Человек получает и спеку, и возвращённый прогон. Повторяемо сколько угодно.

```sql
UPDATE sessions SET status='finished', updated_at=now()
 WHERE id=$1 AND status='active'
```

- `rowcount=0` → сессия уже брошена → **спеку НЕ отдавать**, поднять `StepError`.
- `finish` и `abandon` — взаимоисключающие переходы **одной строки**, а не два независимых
  апдейта. Это и есть настоящая защита; «артефакт-гард» её не заменяет.

## 4. `delete_session` → `abandon_session` — возврат в одной транзакции

**Порядок блокировок: `billing` → `sessions`** — тот же, что в §2 `[C2]`. Обратный порядок
(как было в v1) даёт ABBA-дедлок при одновременных `/start` и `/reset`.

```
BEGIN
  INSERT INTO billing (tg_user_id) VALUES ($1) ON CONFLICT DO NOTHING
  SELECT ... FROM billing WHERE tg_user_id=$1 FOR UPDATE        -- 1) billing первым
  UPDATE sessions SET status='abandoned', refunded=true, updated_at=now()
   WHERE tg_user_id=$1 AND status='active' AND refunded=false   -- 2) потом sessions
   RETURNING id, consumed
  -- rowcount=0 -> нечего возвращать, выходим
  -- consumed='free' -> UPDATE billing SET free_used=free_used-1 WHERE tg_user_id=$1 AND free_used>0
  -- consumed='paid' -> UPDATE billing SET paid_credits=paid_credits+1
COMMIT
```

**Артефакт-гард выброшен `[M2][M3]`.** Он был вреден с двух сторон:
- мёртв там, где нужен: `finish` уже ставит `finished`, значит после выдачи спеки
  `WHERE status='active'` и так ничего не найдёт — критерий 3 закрыт §3, а не гардом;
- вредит там, где срабатывает: `save_artifact` последнего шага проходит **до** `finish`, поэтому
  если `finish` упал на 529 — гард **отказал бы в возврате**. Ровно сценарий инцидента,
  воспроизведённый защитой от него же.
- вдобавок он был нереализуем одним запросом: `$final` зависит от `sessions.version`, а
  `UPDATE` ключуется по `tg_user_id` — версию пришлось бы сначала прочитать `[M3]`.

**Пол `free_used>0` вместо `GREATEST` `[m2]`:** `GREATEST` прячет баг, условие в `WHERE` — нет.
`rowcount=0` на возврате при `consumed='free'` → `log.warning` (значит реестр разошёлся с
биллингом, хотим знать).

**Наблюдаемость возврата `[M4]`:** каждый возврат → `log.info` + Sentry-breadcrumb с
`tg_user_id`/`consumed`. Кап на число возвратов **не ставим** (осознанно), но абуз обязан быть
видимым до того, как станет дорогим.

## 5. Протокол, двойник и оба хендлера `[M1][M6]`

v1 молчал про всё это — hermetic-тесты гоняли бы разошедшийся контракт.

- `db/repo.py:86` — в протоколе `delete_session` → `abandon_session(tg_user_id) -> bool`
  (`True` = прогон возвращён), докстринг с контрактом возврата.
- `db/memory_repo.py:154` — реализовать так же: `consumed`/`refunded`/пол/**сохранять messages**
  (сейчас двойник их удаляет). Тест-паритет: одна и та же последовательность
  reset→refund→reset даёт одинаковый ответ у обоих репо.
- `tg/bot.py:117` (`/reset`) и `tg/bot.py:147` (кнопка) — оба зовут `abandon_session` и говорят
  правду по возвращённому флагу:
  - `True` → «Начали заново — прогон вернулся на счёт.»
  - `False` → «Активной сессии не было.»
  Текст «Сессия удалена» — ложь (сессия жива, транскрипт цел) и, главное, умалчивает о возврате,
  ради которого всё делается.
- **После reset показывать клавиатуру версий `[M-2]`.** v2 предлагал «чистить `pending_version`» —
  это **no-op**: `pop(user_id, "lite")` (`bot.py:177`) срабатывает на каждом тексте, словарь и так
  пуст. Проблема реальна, фикс был фиктивным: человек получил возврат, написал «ок» → `on_text`
  создал сессию на дефолтном `lite`, которого он не выбирал, и **снова списал прогон**.
  Правильно: после успешного возврата reset показывает `version_keyboard()`; `on_text` без сессии
  и без pending-версии **не создаёт** сессию с дефолтом, а показывает клавиатуру.

- **`abandon_session -> bool` не должен врать владельцу `[M-5]`.** Владелец идёт через
  `repo.start_session` → `consumed='none'`; его reset даст `rowcount=1` → `True` → бот скажет
  «прогон вернулся на счёт», хотя биллинг его не касается (критерий 6). `True` возвращаем только
  если billing реально изменился; иначе текст «Начали заново.»

- **Незавершённый ход не должен мутировать брошенную сессию `[M-6]`.** После `abandon` in-memory
  `session.status` в работающем ходе остаётся `'active'`: `save_artifact`/`advance_step`
  (`pg_repo.py:69-102`) продолжают писать в неё. Деньги не текут (`finish` отобьётся), но ход
  живёт ещё 6 итераций. Дёшево: добавить `AND status='active'`, `rowcount=0` → `StepError`.

- **Тесты — это тоже call-site'ы `[M-3]`.** `delete_session` зовут 6 раз в тестах
  (`test_billing.py:21,30`, `test_billing_gate.py:24`, `test_owner.py:33,41,66`), причём трое
  используют его как **«сжечь прогон»** и ждут `NEEDS_PAYMENT`. После замены они станут красными
  **законно** — и если их бездумно «поправить», платный путь останется без единого теста.
  Переписать на `set_status('finished')` (прогон сгорает без возврата). `delete_session` удалить
  из протокола и обоих репо целиком — иначе §0 декларация: метод жив, кто-нибудь позовёт.

- **`Session`/`_row_to_session` `[minor]`:** dataclass (`repo.py:12`) и `_row_to_session`
  (`pg_repo.py:33`) не знают про `consumed`/`refunded` — без них тест-паритет №13 не написать.

## 6. Ретраи: бюджет на ХОД, а не на вызов `[C3][C4]`

v1 давал ~48 минут молчания и до 90 запросов на одно сообщение (5 попыток × 90 с таймаута ×
6 итераций tool-loop, ×3 от SDK-ретраев).

- **SDK-ретраи выключаем: `AsyncAnthropic(max_retries=0)`** — иначе наши попытки умножаются на
  его. Ретрай единственный, наш, и он единственный источник правды.
- **Дедлайн на весь ход** `[C-2]`: `turn.py` создаёт `deadline = monotonic() + turn_budget_s`
  (**120 с**) и передаёт в `claude.turn` (**keyword-only с дефолтом** — иначе краснеют 10
  тестовых двойников `claude.turn`).
  Проверять **перед КАЖДОЙ попыткой**, включая первую в каждой из 6 итераций tool-loop, и
  клампить таймаут: `timeout=min(claude_timeout_s, deadline - monotonic())`; остаток `<=0` →
  `ClaudeOverloaded` без запроса. **Без клампа дедлайн — декларация**: 6 итераций × свежие 90 с
  = 540 с молчания (v2 уменьшил 48 мин до 9, а не убил).
- Паузы `[1,2,4,8,16]` ±25% джиттера, но не дольше остатка дедлайна. (Формулировка v2 «ретрай
  ровно один» была враньём против собственных пяти пауз — читать как «один слой ретраев, наш».)
- **Ретраим:** **весь `5xx`** + `429`, `APITimeoutError`, `APIConnectionError`,
  **и `asyncio.TimeoutError`/`TimeoutError`** `[C4]` — последние поднимает `asyncio.wait_for`
  (`client.py:31`), это НЕ `anthropic.APITimeoutError`.
  ⚠️ `[M-1]` v2 брал только `429/529` → `500/502/503` не ретрились бы **ни разу** (сейчас их
  глушит SDK, который мы выключаем). Тихая регрессия — поэтому весь 5xx.
- **Не ретраим:** `4xx` кроме `429` — поднимаем сразу.
- `retry-after` уважаем; если он больше остатка дедлайна — **не ждём**, сразу `ClaudeOverloaded`
  (долбить нельзя, врать про ожидание тоже) `[m6]`.
- `asyncio.wait_for` остаётся вокруг **одной** попытки.

## 7. Честный текст + не бросать человека `[m5]`

```
except ClaudeOverloaded:
    log.error("anthropic overloaded", exc_info=True)   # [M-4] error, НЕ warning
    "Claude сейчас перегружен — это на моей стороне, не у тебя.
     Твой ответ я сохранил: напиши «продолжай» через пару минут."
except Exception:
    log.exception("turn failed")
    "Упс, что-то сбойнуло на моей стороне. Попробуй ещё раз."
```

- **`log.error`, а не `warning` `[M-4]`.** `sentry_sdk.init` (`observability.py:61`) идёт без
  явной `LoggingIntegration` → дефолт `event_level=ERROR` → `log.warning` дал бы **breadcrumb,
  а не issue**, и под алёрт `console-critical-errors` не попал бы. То есть §7 в v2 делал ровно
  то, от чего сам предостерегал: ослеплял нас на классе, ради которого всё затевалось.
- **Промежуточный сигнал `[инвариант #6]`:** на первой паузе — однократное «Claude тупит, жду…».
  120 с молчания — это тоже «бросить человека». ⚠️ Транспорт: `ClaudeClient` не видит
  `on_notice` — прокидывать колбэком **из `turn.py`**, а не тянуть aiogram в `llm/`.
- **Детектор из `CLAUDE.md` придётся уточнить `[minor]`.** Совет «напиши „продолжай“» делает
  «два `user` подряд» **штатной** картиной при каждой перегрузке (реплика человека в БД есть,
  assistant-строки нет). Петли не будет (`normalize_history` склеит), но дешёвый детектор начнёт
  ложно срабатывать → писать маркер-assistant `[перегрузка, ответа не было]`, чтобы транскрипт
  остался честным (инвариант #5).

## Риски

- Дедлок → единый порядок `billing → sessions`, зафиксирован комментарием в обоих методах.
- Дыра C1 → условный `finish` + тест 10.
- `max_retries=0` меняет поведение на микро-морганиях (раньше их глушил SDK) — наш backoff
  начинается с 1 с, покрывает.
- Миграция на пустой таблице мгновенна; при непустой — см. бэкфилл `[m3]`.
