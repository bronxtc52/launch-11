# ТЗ — launch-11 бот, Фаза 3 (монетизация Telegram Stars)

**Статус:** черновик на подтверждение
**Класс:** L, **high-risk (деньги)** — authz/идемпотентность обязательны
**Дата:** 2026-07-15 · **Опирается на:** Фазы 1-2 (петля, версии, ADR, PgRepo)

## Цель

Монетизация по модели Marga: **1 бесплатный прогон** на пользователя → дальше платно
**звёздами Telegram (XTR)**. Снять Phase-1 access-gate (allowlist) и заменить **биллинг-гейтом**.
Списание — **при старте сессии (резерв)**: чтобы начать НОВУЮ сессию нужен слот
(`free_used < FREE_RUNS`) или оплаченный кредит; он списывается сразу. Возобновление той же
сессии — бесплатно. Так ограничивается стоимость Claude (нельзя бесконечно жечь диалог даром).

## User stories

1. Новый пользователь: первый прогон бесплатный — сразу начинает без оплаты.
2. Исчерпав бесплатный слот, при попытке начать новый прогон получаю **счёт на XTR**; оплатив,
   получаю кредит на следующий прогон.
3. Оплата проходит через нативные Telegram Stars (без внешнего провайдера).
4. Двойная доставка `successful_payment` (ретрай Telegram) не начисляет кредит дважды.

## Модель данных (миграция 003)

- `billing(tg_user_id PK, free_used INT DEFAULT 0, paid_credits INT DEFAULT 0, updated_at)`
  — одна строка на пользователя, источник истины для права на прогон.
- `payments(charge_id TEXT PK, tg_user_id BIGINT, stars INT, created_at)` — **PK по
  `telegram_payment_charge_id`** = дедуп платежа (идемпотентность начисления).

## Логика (вся на сервере, в транзакции — уроки KB)

- **`try_consume_entitlement(tg_user_id, free_runs) -> bool`** (одна транзакция, `SELECT … FOR
  UPDATE` строки billing или upsert):
  - `free_used < free_runs` → `free_used++`, вернуть True;
  - иначе `paid_credits > 0` → `paid_credits--`, вернуть True;
  - иначе вернуть False (нет права → отправить счёт).
- **`grant_paid_credit(charge_id, tg_user_id, stars) -> bool`** (одна транзакция):
  `INSERT INTO payments(charge_id…) ON CONFLICT (charge_id) DO NOTHING`; если строка вставлена
  (новый платёж) → `paid_credits++`, вернуть True; если конфликт (дубль) → ничего, вернуть False.
- **Порядок гейта (authz):** право на прогон проверяется **ДО** создания сессии и **ДО** любого
  вызова Claude — и в text-first потоке, и при клике по кнопке версии.

## Telegram Stars

- **Счёт:** `sendInvoice(currency="XTR", prices=[LabeledPrice(label, STARS_PRICE)],
  provider_token="")` (для Stars provider_token пустой). Цена/лейбл — из config.
- **`pre_checkout_query`** → `answer(ok=True)` (в пределах 10 с).
- **`successful_payment`** → `grant_paid_credit(...)` + уведомить пользователя; идемпотентно.

## Config

`FREE_RUNS=1`, `STARS_PRICE` (напр. 100), `STARS_LABEL="Прогон пайплайна"`. Опционально
`BETA_ALLOWLIST` — если задан непустым, дополнительно ограничивает (kill-switch на раскатку;
по умолчанию пусто = публично, гейтит только биллинг).

## Границы — что НЕ делаем в Фазе 3

- ❌ Прод-деплой ACA и создание Azure-ресурсов (красная зона, отдельно с «ок»).
- ❌ Возврат/refund Stars, промокоды, подписки, пакеты >1 кредита за платёж (кредит = 1 прогон).
- ❌ Возврат слота за брошенную сессию (осознанно: списание при старте, см. развилку).
- ❌ Реальный платёж в тестах — Telegram API мокается; живая оплата — ручная приёмка.

## Критерии приёмки (проверяемые да/нет)

1. Новый пользователь: `try_consume_entitlement(u, free_runs=1)` → True, `free_used=1`.
2. Тот же пользователь без кредитов, второй раз → False; `free_used` остаётся 1.
3. `grant_paid_credit("ch1", u, 100)` → True, `paid_credits=1`; затем `try_consume` → True,
   `paid_credits=0`.
4. **Идемпотентность:** повторный `grant_paid_credit("ch1", …)` → False; в `payments` одна строка
   с `ch1`; `paid_credits` не изменился.
5. **Гейт ДО Claude:** пользователь без права (`free_used≥1`, `paid_credits=0`) шлёт текст →
   бот отправляет счёт, **Claude не вызывается**, сессия не создаётся (мок Claude: 0 вызовов).
6. `pre_checkout_query`-хендлер отвечает `ok=True`.
7. `successful_payment`-хендлер вызывает `grant_paid_credit` и уведомляет пользователя.
8. Счёт строится с `currency="XTR"`, пустым `provider_token`, ценой из config.
9. **Гонка/идемпотентность (PG):** два конкурентных старта (текст ИЛИ клик по версии) для
   свежего пользователя → создаётся ОДНА сессия и списывается ОДНО право (итог `free_used=1`,
   не 2). Списание привязано к созданию сессии в одной транзакции.
10. **Персистентность (PG):** миграция 003 создаёт `billing`/`payments`; состояние переживает
    реконнект (новый PgRepo видит `free_used`/`paid_credits`).
11. **Валидация платежа:** `successful_payment` с `currency≠"XTR"` ИЛИ `total_amount≠STARS_PRICE`
    → кредит НЕ начисляется (нет строки в `payments`, `paid_credits` не растёт).
12. **Привязка намерения:** `invoice_payload` содержит id пользователя и проверяется при
    начислении; чужой/битый payload → не начислять.

Тесты 1-8 — hermetic (InMemoryRepo эмулирует ту же транзакционную семантику). 9-10 — integration
на живом PG. Живая оплата в Telegram — ручная приёмка.

## Приёмка сдачи

PR. Прод-деплой — не в этой фазе.
