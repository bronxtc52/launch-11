# ТЗ — launch-11 бот, Фаза 1 (петля end-to-end)

**Статус:** черновик на подтверждение
**Класс задачи:** M (новый сервис, но узкий срез; чувствительных данных нет)
**Дата:** 2026-07-15

## Цель

Рабочая **петля end-to-end**: пользователь в Telegram запускает бота → проходит
**Lite-версию** пайплайна Сейсембая в виде сократического диалога → на выходе
получает одну собранную `spec.md` как **документ** (`sendDocument`).

Это фундамент: доказать связку «свободный LLM-диалог внутри шага + детерминированный
переход между шагами + сборка markdown-документа». Full 11 шагов и биллинг —
следующие фазы, но архитектура закладывается под них сразу (шаги — декларативные данные).

## User stories

1. Как пользователь, я пишу боту `/start`, он объясняет что будет и спрашивает идею.
2. Как пользователь, я веду свободный диалог; бот задаёт сократические вопросы и
   предлагает 2-3 варианта формулировки кнопками (А/Б/В/свой вариант).
3. Как пользователь, когда шаг доработан, я вижу «✅ Шаг N зафиксирован» и кнопку «Дальше →».
4. Как пользователь, в любой момент я жму «Прогресс» и вижу, на каком я шаге.
5. Как пользователь, после последнего шага я получаю `<slug>-spec.md` файлом в чат.
6. Как пользователь, если я вернулся через день и снова написал `/start`, бот
   помнит мой прогресс и предлагает продолжить, а не начинает заново.

## Архитектура (Фаза 1)

```
Telegram ⇄ aiogram 3 (long-polling) ⇄ orchestrator ⇄ Claude Sonnet 5 (tool-use)
                                          │
                                      Postgres (sessions / artifacts / messages)
                                          │
                              assemble → <slug>-spec.md → sendDocument
```

- **Мозг:** Claude Sonnet 5, system prompt = адаптация `claude-ai/SKILL.md` (методика,
  стиль, антипаттерны) + активная инструкция текущего шага (прогрессивно, экономит контекст).
- **Детерминизм поверх диалога** — через tool-use. Инструменты в Фазе 1:
  - `save_artifact(step_id, markdown)` — фиксирует артефакт шага, продвигает `current_step`.
  - `set_version(lite|full|spec_only)` — в Фазе 1 принимается только `lite`.
  - `finish()` — доступен только когда все Lite-шаги закрыты → сборка + `sendDocument`.
- **Шаги — декларативные данные** (`steps.py`: id, zone, title, goal, instruction).
  Lite-набор (Фаза 1), 4 контентных шага + finish:
  - `L1` — Идея + Northern Star + Vision Story (метафора) [объединённые шаги 1-2]
  - `L2` — Vision Contract: проблема / аудитория / решение / **границы (≥5 «не делаем»)** / успех
  - `L3` — Короткая архитектура: стек + хостинг (1-2 варианта, выбор пользователя)
  - `L4` — Контекст для сборки: Northern Star, целевой пользователь, «не делаем», задачи списком
  - `finish` — сборка `spec.md`
- **Санитайзер:** вендорим `md_to_telegram_html` из Avicenna (`apps/bot/app/util.py`).
  Порядок строгий: `html.escape` первым → `**`→`<b>` → `##`→`<b>` → списки `•` →
  `*`/`_`→`<i>` → `` ` ``→`<code>`. `parse_mode=HTML`. Чанковать >4096 **по границам строк**.
- **Персистентность:** всё состояние в Postgres; in-memory только кэш. Сессия переживает рестарт.

## Данные (Postgres, Фаза 1)

- `sessions(id, tg_user_id, slug, version, current_step, status, created_at, updated_at)`
  — одна активная сессия на пользователя (Фаза 1); `status ∈ {active, finished, aborted}`.
- `artifacts(id, session_id, step_id, markdown, updated_at)` — UNIQUE(session_id, step_id),
  повторный `save_artifact` того же шага = **перезапись, не дубль** (идемпотентность).
- `messages(id, session_id, role, text, created_at)` — транскрипт для контекста Claude.

## Границы — что НЕ делаем в Фазе 1

- ❌ Биллинг / Telegram Stars (Фаза 3) — бот бесплатный и без лимитов в этой фазе.
- ❌ Full 11 шагов (Фаза 2) — только Lite.
- ❌ ADR / BACKLOG-файлы / zip-каркас репо в выходе — только одна `spec.md`.
- ❌ Web-research (шаги 4-5) — Lite их пропускает.
- ❌ Прод-деплой ACA и создание Azure-ресурсов (красная зона, отдельным шагом с «ок»).
  Фаза 1 сдаётся как код + локальный `docker-compose` (бот + Postgres) + зелёные тесты.
- ❌ Мультипродукт на пользователя, экспорт истории, админка.

## Стек

Python 3.12 · aiogram 3 · anthropic SDK (Sonnet 5) · asyncpg/psycopg + Postgres 16 ·
Sentry SDK · `.env` из KV через `fetch-env.sh` (локально; ACA-секреты — позже).
Long-polling: **без ingress, single-replica** (грабля: ingress → startup-проб → крэш-луп).

## Критерии приёмки (проверяемые, да/нет)

1. **Санитайзер:** `md_to_telegram_html("**жирный**")` → содержит `<b>жирный</b>`;
   `md_to_telegram_html("ЛПНП <3,0")` не содержит сырого `<` (экранирован в `&lt;`).
2. **Чанковка:** сообщение из 200 строк по 50 символов режется на части ≤4096,
   ни один тег `<b>…</b>` не разорван между частями.
3. **Старт сессии:** новая сессия создаётся с `version=lite`, `current_step=L1`, `status=active`.
4. **Переход:** `save_artifact("L1", md)` сохраняет артефакт и выставляет `current_step=L2`;
   попытка `save_artifact("L3", …)` при `current_step=L1` отклоняется (нельзя перескочить).
5. **Идемпотентность:** два `save_artifact("L1", …)` подряд → в `artifacts` одна строка для L1
   (перезапись), `current_step` не уезжает дальше положенного.
6. **finish-гейт:** `finish()` при незакрытых шагах → отказ; при всех закрытых → возвращает
   собранный markdown со всеми артефактами в порядке L1→L4 и заголовком по slug.
7. **Диспетчер tool-use:** ответ Claude с блоком `save_artifact` → оркестратор вызывает
   сохранение и отдаёт подтверждение; неизвестное имя инструмента → безопасная ошибка (не краш).
8. **Персистентность:** после пересоздания процесса (новый коннект к той же БД) `/start`
   для существующего `tg_user_id` со `status=active` возвращает текущий шаг, не сбрасывает прогресс.

Тесты — на уровне оркестратора/санитайзера/сборки; Telegram I/O и вызовы Claude мокаются.
Живой прогон с реальным ботом+Claude — ручная проверка после зелёных юнитов (evidence в отчёте).

## Приёмка сдачи

- Все 8 критериев зелёные юнит-тестами (мок Claude/Telegram).
- `docker-compose up` поднимает бот + Postgres; `/start` в реальном тест-боте доводит
  до `spec.md` файлом (скрин/лог как evidence).
