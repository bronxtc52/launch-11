# ТЗ — launch-11 бот, Фаза 2 (Full 11 шагов + Spec-only + версии)

**Статус:** черновик на подтверждение
**Класс:** M (данные + узкие точечные правки; архитектура версионная с Фазы 1)
**Дата:** 2026-07-15 · **Опирается на:** Фаза 1 (петля, `PIPELINES[version]`, tool-use, PgRepo)

## Цель

Поверх готовой Lite-петли добавить **Full-версию** (все 11 шагов методики Сейсембая) и
**Spec-only** (только шаги 8-11), с **выбором версии при старте**. Реестр пайплайнов уже
версионный — задача в основном декларативная (данные шагов) + точечные правки инструмента
`set_version`, сборки `spec.md` и поддержка **ADR**.

## User stories

1. Как пользователь, при `/start` я выбираю версию кнопками: **Full (11 шагов) / Lite / Spec-only**.
2. Как пользователь на Full, я прохожу все 11 шагов (Смысл 1-7 → Bridge 8-10 → Реализация 11)
   с сократическими вопросами на каждом.
3. Как пользователь, на шагах 6 и 8 значимые решения фиксируются как **ADR**, и они попадают
   в итоговую `spec.md` отдельным разделом.
4. Как пользователь на Spec-only (у меня уже есть Vision), я начинаю сразу с архитектуры (шаг 8).
5. Как пользователь, финальная `spec.md` для Full содержит все шаги в порядке + ADR.

## Объём

1. **`FULL_STEPS`** — 11 шагов (`F1..F11`) с инструкциями из `references/step-01..11`.
2. **`SPEC_STEPS`** — срез шагов 8-11 Full (переиспользуют те же `Step`, ids `F8..F11`).
3. **Выбор версии** при `/start` — inline-кнопки Full/Lite/Spec-only; сессия создаётся с выбранной.
4. **`set_version`** — enum расширен до `["lite","full","spec_only"]`; диспетчер принимает все три.
   Смена версии разрешена **только на пустой сессии** (нет ни одного артефакта) — иначе отказ
   (не осиротить уже сохранённые артефакты); при смене `current_step` сбрасывается на первый шаг версии.
5. **ADR:** новый инструмент `create_adr(title, markdown)`; таблица `adrs` (миграция `002`);
   методы репозитория (InMemory + Pg); диспетчер; сквозная нумерация ADR по сессии.
6. **Сборка `spec.md`**: идёт по `PIPELINES[version]` (уже так); для версий с ADR добавляется
   раздел `## Решения (ADR)` со всеми ADR по порядку.

## Границы — что НЕ делаем в Фазе 2

- ❌ Биллинг / Telegram Stars (Фаза 3).
- ❌ Прод-деплой ACA и создание Azure-ресурсов (красная зона, отдельно).
- ❌ Кнопки А/Б/В выбора формулировок (UX-полировка, отдельная задача).
- ❌ zip-каркас репо / раздельные файлы артефактов — выход по-прежнему одна `spec.md`.
- ❌ Web-research как автоматизация на шаге 5 — бот ведёт диалог, поиск не автоматизируем здесь.

## Критерии приёмки (проверяемые да/нет)

1. `PIPELINES` содержит `full` (11 шагов) и `spec_only` (4 шага); `step_ids("full")` длиной 11,
   `step_ids("spec_only")` = `["F8","F9","F10","F11"]`.
2. Выбор версии: создание сессии с `version="full"` → `current_step` = первый шаг Full (`F1`);
   с `version="spec_only"` → `current_step="F8"`.
3. `set_version` на **пустой** сессии: `lite→full` меняет `version` и сбрасывает `current_step`
   на `F1`; возвращает успех.
4. `set_version` на сессии **с артефактом**: отклоняется, `version`/`current_step` не меняются.
5. Схема инструмента `set_version` имеет enum `["lite","full","spec_only"]`; `dispatch` принимает
   каждую из трёх на пустой сессии.
6. `create_adr("выбор БД", "...")` → ADR сохранён с номером 1; второй → номер 2; `get_adrs`
   возвращает оба по порядку.
7. `assemble_spec` для Full со всеми 11 артефактами + 2 ADR: содержит все шаги в порядке `F1..F11`
   и раздел `## Решения (ADR)` с обоими ADR.
8. `assemble_spec` для `spec_only`: содержит только шаги `F8..F11`, без шагов 1-7.
9. finish-гейт Full: `can_finish` True только когда есть артефакты всех 11 шагов (ADR
   не обязательны для finish); при неполном наборе `finish` отклоняется.
10. **Персистентность (PG, integration):** миграция `002` создаёт `adrs`; `create_adr` переживает
    реконнект к той же БД (ADR читаются новым PgRepo).

Тесты — hermetic (InMemoryRepo) для 1-9; integration на живом PG для 10. Telegram version-select
UI (кнопки) — ручная проверка, логика выбора версии тестируется в сервисе.
