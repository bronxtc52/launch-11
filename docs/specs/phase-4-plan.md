# План — launch-11 бот, Фаза 4 (один вопрос + валидация ответа)

**ТЗ:** `phase-4-dialogue.md` · **Класс:** L, не high-risk · **Код:** Sonnet

## Принцип

Тот же, что держит весь бот: **свободный диалог внутри шага, детерминизм — через tool-use**.
Вопрос перестаёт быть «свободным текстом» и становится вызовом инструмента, который бот
рендерит сам. Оценка ответа — тоже инструмент, без него модель не идёт дальше.

## Ключевые решения

1. **Order-gate в turn-сервисе, не в диспетчере.** Порядок вызовов — свойство хода диалога;
   сервис владеет циклом. При открытом `current_question` первый не-`assess_answer` tool
   получает `tool_result` «сперва оцени ответ» и НЕ исполняется (состояние не трогается).
2. **offtopic обрабатывает бот, не модель.** Бот сам повторяет сохранённый вопрос (детерминизм:
   формулировка не «переизобретается» моделью) и завершает ход.
3. **`partial` блокирует `save_artifact`** на уровне оркестратора — нельзя проскочить шаг
   с неполным ответом (это и просил пользователь).
4. **Ретрай контракта вывода — ровно один**, потом текст уходит как есть: дисциплина не должна
   превращаться в отказ обслуживания. Нарушение логируется.
5. **`current_question` в БД**, не в памяти — вопрос переживает рестарт (Фаза 1 инвариант).

## Правки по совету моделей (v2, вердикт request_changes → приняты)

Совет (5 ролей, $0.87) дал 13 находок и вскрыл дыру в замысле: я принуждал **канал**, но не
**содержимое**. Принято:

- **[HIGH product_risk-1] Валидировать текст вопроса детерминированно.** Без этого модель
  положит «1. … 2. … 3. …» ВНУТРЬ `ask_question(question=…)` и свалка просто переедет в
  инструмент. Диспетчер отклоняет `ask_question`, если в `question`+`preamble`: >1 знака `?`,
  нумерованный/маркированный список, перевод строки со вторым вопросом, длина > лимита.
  **Это ядро фикса, а не деталь.**
- **[HIGH architect-2] Инспектировать ВЕСЬ ответ модели до отправки чего-либо.** Текущий цикл
  шлёт `turn.text` сразу, до гейтов → свалка уйдёт раньше проверки. `run_user_turn`
  перестраивается: получить ответ → проверить контракт → только потом рендерить.
- **[HIGH correctness-2] `ask_question` — терминален для хода.** После него tool-calls в том же
  ходу не исполняются: вопрос задан, ждём человека (иначе модель «сама себе ответит» и сохранит
  артефакт).
- **[HIGH product_risk-2] Fail-closed после провала ретрая.** Нарушающий текст НЕ отправляем
  (иначе дисциплина обходится упрямством модели): извлекаем **первый** вопрос и шлём только его;
  если извлечь нечего — повторяем `current_question` / нейтральный nudge. Нарушение логируется.
- **[MEDIUM product_risk-3] Контракт шире, чем «нет `?`».** Просьба без знака вопроса
  («Расскажи про пользователя.») — тоже вопрос. Правило: **пользовательская проза без единого
  tool-call, когда ожидается ответ, = нарушение** → ретрай.
- **[MEDIUM product_risk-5] Выход из вопроса — `/skip`.** Человек не должен застревать: `/skip`
  снимает `current_question` и разблокирует шаг (методика прямо допускает «соскочить»).
- **[MEDIUM architect-4] CHECK-констрейнт на verdict** в миграции 004 + валидация в репо.

**Отклонено (с обоснованием):**
- *architect-3 (отдельная фаза оценки отдельным вызовом Claude)* — удваивает вызовы на каждую
  реплику (деньги+латентность). Эффект достигается order-gate + инспекцией всего ответа за
  **один** вызов.
- *product_risk-4 (независимый верификатор вердикта)* — accept-risk: продукт помогает самому
  пользователю, состязательной стороны нет; гейт от честных ошибок, не от атаки.
- *performance-1 (таймаут на вызов Claude)* — **уже реализовано** в `llm/client.py`
  (`asyncio.wait_for` + кап ретраев); файла не было в packet.

### Новые критерии (в ТЗ добавлены 11-15)

11. `ask_question` с >1 вопросом / нумерованным списком → `ok=False`, пользователю не уходит.
12. `ask_question` терминален: tool-calls после него в том же ходу не исполняются.
13. Проза без tool-call при ожидаемом ответе → ретрай; после провала — исходный текст НЕ
    отправляется (fail-closed), уходит извлечённый первый вопрос / повтор текущего.
14. `/skip` очищает `current_question` и снимает блокировку `save_artifact`.
15. `verdict` вне enum отклоняется (CHECK в БД + репо).

## Риски / грабли

- **Ломается сигнатура `run_user_turn`** (добавляется `on_question`) → обновить тесты Фаз 1-3
  (они уже адаптировались под `version=` в Фазе 3 — тот же приём, прозрачно отметить).
- **Модель может звать `ask_question` и параллельно писать вопрос текстом** → рендерим вопрос
  ТОЛЬКО из инструмента; свободный текст с `?` при наличии `ask_question` не ретраим (уже
  по контракту), но и не дублируем вопрос.
- **`assess_answer` на первом сообщении шага** (вопроса ещё нет) — не требуется: gate включается
  только при непустом `current_question`.
- **Дрейф инструкций шагов обратно** — закрывается регресс-тестом (крит. 8), а не бдительностью.
