# План — launch-11 бот, Фаза 2 (Full 11 + Spec-only + ADR)

**ТЗ:** `phase-2-full11.md` · **Класс:** M, не high-risk · **Модель кода:** Sonnet

## Ключевые решения

1. **Данные, не код.** Реестр `PIPELINES` из Фазы 1 уже версионный. Добавляем
   `FULL_STEPS` (F1..F11) и `SPEC_STEPS = FULL_STEPS[7:11]` (переиспользуют те же `Step`,
   ids `F8..F11`). Оркестратор/сборка/finish-гейт уже generic — трогать не нужно.
2. **Смена версии — только на пустой сессии.** `set_version` в диспетчере проверяет
   `get_artifacts(session_id)` пуст → меняет `version` + ресет `current_step` на первый
   шаг новой версии (атомарно через новый метод репо `set_version(session_id, version, first_step)`);
   иначе безопасный отказ. Не осиротить артефакты (ids разных версий не совпадают).
3. **ADR как отдельная сущность.** Новый инструмент `create_adr(title, markdown)`; таблица
   `adrs(session_id, n, title, markdown)` (миграция `002`), сквозная нумерация `n` по сессии.
   Сборка добавляет раздел `## Решения (ADR)` когда ADR есть. ADR не обязательны для finish.
4. **Step F11 (handoff)** адаптирован под бота: инструкция = подвести итог + предложить
   `finish()`; заливка на VPS остаётся текстовой рекомендацией.

## Тест-стратегия

- Hermetic (InMemoryRepo) — крит. 1-9. Регрессия Фазы 1 остаётся зелёной (та же петля,
  Lite не тронут).
- Integration (живой PG) — крит. 10 (миграция 002 + adrs через реконнект).
- Version-select кнопки в Telegram — ручная проверка; логика выбора версии (создание сессии
  нужной версии) тестируется через сервис/оркестратор.

## Риски / заметки

- **ids версий не пересекаются** (L*, F*) — переключение版ии на пустой сессии безопасно;
  на непустой запрещено. Проверяется крит. 3-4.
- **Регрессия сборки:** `assemble_spec` меняет сигнатуру (добавляется ADR) — сохранить
  обратную совместимость дефолтом `adrs=None`, чтобы тесты Фазы 1 не падали.
- ADR-нумерация сквозная по сессии — гонку игнорируем (одиночный пользователь, один поток
  диалога); на PG — `n` вычисляется в транзакции (MAX+1) как у `save_and_advance`.
